# Immo Eliza — Price Prediction API & Web App

Predicts the sale price of a property in Belgium from its characteristics
(surface, location, condition, energy score...), using 5 regression models
trained on a cleaned real estate dataset. Built as a FastAPI backend and a
Streamlit frontend, deployed as two separate services.

## Live demo

- **API** (Render): https://immo-eliza-deployment-zla0.onrender.com
- **Web app** (Streamlit Community Cloud): https://immo-eliza-deployment-hwgvzpv33iu3tdw8nxdrfs.streamlit.app/

> The API is hosted on Render's free tier, which spins down after 15 minutes
> of inactivity. The first request after a while can take up to ~50 seconds
> to wake it back up — this is expected, not a bug.

## Architecture

```
                        ┌─────────────────────┐
                        │   Streamlit app      │
                        │  (Streamlit Cloud)   │
                        │  model dropdown,      │
                        │  color-coded form,    │
                        │  map, importance chart│
                        └──────────┬───────────┘
                                   │ HTTPS (JSON)
                                   ▼
                        ┌─────────────────────┐
                        │     FastAPI API       │
                        │      (Render)         │
                        │  app.py + predict.py  │
                        └──────────┬───────────┘
                                   │ loads at startup
                                   ▼
                        ┌─────────────────────┐
                        │   models/ artifacts   │
                        │  1 registry + 5       │
                        │  fitted pipelines      │
                        └─────────────────────┘
```

The frontend and the backend are two independently deployed services that
only communicate over HTTP — the Streamlit app never loads a model
directly, it only calls the API.

## Models

5 regression models were trained and compared on the same train/test split:

| Model         | R² (test) |
|---------------|-----------|
| XGBoost       | 0.7391    |
| Random Forest | 0.6678    |
| Ridge         | 0.6235    |
| SVM (SVR)     | 0.5780    |
| Decision Tree | 0.4629    |

XGBoost and Random Forest outperform the others here, which is a typical
result for tabular data: boosting/bagging tree ensembles usually capture
non-linear interactions between features (surface, location, condition...)
better than a single tree or a linear model. Random Forest and XGBoost also
apply a stricter anti-overfitting guard during tuning than the other three,
trading a bit of raw test performance for a smaller train/test gap.

RMSE and MAE for every model are available live from the API (`GET /models`),
since they can shift slightly between retraining runs.

## Model artifacts

The files in `models/` are generated by a companion training script (not
part of this repository, since it depends on the raw training CSV) that:
1. trains all 5 models with their own tuning strategy,
2. computes each model's own feature-importance ranking (grouped by
   category — e.g. `region_Brussels`/`region_Flanders`/`region_Wallonia`
   collapsed into one `region` score), stored as `None` for a model whose
   estimator doesn't expose one (e.g. an SVM with a non-linear kernel),
3. saves one lightweight pipeline per model plus a single shared
   `models_registry.joblib` — no raw training data is bundled into these
   files.

## API reference

| Method | Route                | Description                                                   |
|--------|----------------------|-----------------------------------------------------------------|
| GET    | `/`                  | Health check, returns `"alive"`                                |
| GET    | `/models`            | List of models with display name + test scores (R², RMSE, MAE) |
| GET    | `/feature-importance`| `?model=<name>` — that model's grouped feature importance, or `null` |
| POST   | `/predict`           | `{"model": "<name>", "data": {...}}` → `{"prediction": float\|null, "status_code": int}` |

Every field in `data` is optional — missing numeric values are handled by
the pipeline's own median imputer, so a partial listing still returns a
prediction.

Example request:
```bash
curl -X POST https://immo-eliza-deployment-zla0.onrender.com/predict \
  -H "Content-Type: application/json" \
  -d '{
        "model": "xgboost",
        "data": {
          "living_area_m2": 120,
          "bedrooms": 3,
          "province": "Brussels Capital Region",
          "region": "Brussels"
        }
      }'
```

Interactive docs (Swagger, auto-generated): `/docs` on the API URL above.

## Running locally

**API:**
```bash
pip install -r requirements.txt
uvicorn app:app --reload
```
→ http://127.0.0.1:8000/docs

**Streamlit app:**
```bash
cd Streamlit
pip install -r requirements.txt
streamlit run streamlit_app.py
```

**Docker (what Render runs):**
```bash
docker build -t immo-eliza-api .
docker run -p 8000:8000 immo-eliza-api
```

## Tech stack

- **Modeling**: scikit-learn (Ridge, Decision Tree, Random Forest, SVR),
  XGBoost
- **API**: FastAPI, Uvicorn, Pydantic
- **Frontend**: Streamlit, Altair
- **Deployment**: Docker + Render (API), Streamlit Community Cloud (frontend)

## Known limitations

- Render's free tier cold start delays the first request after inactivity.
- Feature-importance ranking depends on what the underlying estimator
  exposes (`feature_importances_` or `coef_`); a model trained with an
  estimator that exposes neither (e.g. SVR with a non-linear kernel)
  returns `null` for that route instead of a ranking.
- Predictions are single-point estimates (no confidence interval).
